Debezium MySQL Snapshot 流程解析

发表于 2021-07-28 更新于 2021-08-08

Debezium Mysql Snapshot 对所选表及 Schema 表 [ DDL ] 的全量历史数据进行了读取，该流程是在 Mysql 一致性备份流程 [ Mysql Dump ] 基础上扩展得来，主要的技术点：Transaction、Repeatable Read、FTWRL [ FLUSH TABLES WITH READ LOCK ] 、Binlog 位点。

MySQL 一致性备份

对于以 InnoDB 为存储引擎的 MySQL 数据库，MySQL Dump 可以通过添加 –single-transaction 参数来保证备份后数据的一致性（处于同一时刻），其主要通过将事务 [ Transaction ] 的隔离级别设置为 Repeatable Read，对当前时刻的数据库进行快照，保证该事务任何时刻的表读取均和快照点数据一致。 由于 Repeatable Read Transaction 不适用于 Binlog 位点查询，MySQL Dump 通过添加 –master-data 参数来获得一致性 Binlog 位点，其主要使用 FTWRL 技术冻结数据库的一切写入操作，保障 Binlog 位点和事务的一致性，主要步骤如下：

Debezium Snapshot 流程

  • 使用全局读锁执行 snapshot 的流程，参考上述 MySQL FTWRL 存在的风险，当遇到了Kafka Connect 日志中有 unable to flush and acquire global read lock 或者报错提示 User does not have the 'LOCK TABLES' privilege 说明其 FTWRL 执行异常，可能会导致数据库长时间不可写。

    Step Action
    1 获取全局锁 [ FLUSH TABLES WITH READ LOCK ]，全库不可写。锁持续到完成 schema 表 snapshot结束，保障 schema、位点、事务的一致性。
    2 启动 repeatable read 事务确保非 shcema 表数据依次抽取时的一致性。
    3 读取当前的 binlog 位点。
    4 读取 schema 数据到内存并写入 schema history topic。
    5 释放全局锁，此时库可进行写。通常锁库的总时长很短，几秒内即可完成。
    6 根据设置，选择是否将 schema 数据写入到 schema change topic。
    7 依次扫描非 shcema 表数据，并写入相应的 serverName.databaseName.tableName 表中。
    8 提交事务。
    9 在 connector offsets 中记录完成 snaoshot。

  • 使用表级锁执行 snapshot 的流程，当无法获取全局锁（一些 RDS 或者特殊的数据库不允许使用全局锁），全局锁 snapshot 流程会退化为表级锁 snapshot 流程 （要求赋予用户 LOCK_TABLES 权限）。

    Step Action
    1 启动 repeatable read 事务确保非 shcema 表数据依次抽取时的一致性。
    2 获取表级锁 [ FLUSH TABLES xxxxxx WITH READ LOCK ]。
    3 读取当前的 binlog 位点。
    4 读取 schema 数据并写入 schema history topic。
    5 根据设置，选择是否将 schema 数据到内存并写入到 schema change topic。
    6 依次扫描非 shcema 表数据，并写入相应的 serverName.databaseName.tableName 表中。
    7 提交事务。
    8 释放表级锁，此时库可进行写。通常锁库的时间较长，几乎持续了整个 snapshot 时长。
    9 在 connector offsets 中记录完成 snaoshot。
