Hudi索引机制探析

发表于 2023-07-19 更新于 2023-07-27 分类于 大数据

Hudi 通过索引机制提供高效的 Upsert/Delete 操作，它维护着 HoodieKey 和 FileID 之间的映射关系。在 Upsert 写时，利用索引快速确认记录是新增（I）还是更新（U）以及需要更新文件的位置，以消除不必要的读写。

Overview

索引具体过程是将给定的 HoodieKey (RecordKey + PartitionPath) 与 FileID（文件组ID） 建立唯一映射。这种映射关系在数据第一次写入文件后保持不变。所以，一个 文件组 包含了一组记录的所有版本。

下图为例，白色是基本文件，黄色是更新数据。左边为 Hudi 模型，右边为 Hive ACID 模型。在不具备索引的 Hive ACID 模型中，需要全局尝试合并，把增量文件和每一个基本文件进行合并(100 + 25 * 8) * 4 = 1200MB，在具备索引的 Hudi 模型中，仅需按照FileGroup分组后再合并100 * 4 + 25 * 8= 600MB。可以发现索引大大降低了了合并的成本。

目前 Hudi 支持以下四种索引，又分为两种类型

索引选项 原理 优点 缺点
Bloom 每个 Parquet 文件维护一个 Bloom Filter，在索引查找阶段，把所有可能更新的分区的文件的 Bloom Filter 加载进来，用来判断 Record Key 是否存在 效率高，不依赖外部系统 假阳性问题，还需回溯原文件再查找一遍
Simple 在索引查找阶段，把所有可能更新的分区的文件的 HoodieKey 和 FileID 列加载进来，使用 left outter join 判断 Record Key 是否存在 实现最简单，无需额外的资源 性能比较差
HBase 维护每一个 Record Key 的 Partition Path 和 File Group，在索引查找阶段所有task 向 HBase 发送 Batch Get 请求，获取 Record Key 的Mapping 信息。 对于小批次的 keys，查询效率高 需要外部的系统，增加了运维压力
Custom 自定义索引

全局索引：全局索引在全表的所有分区范围下强制要求键的唯一性，也就是确保对给定的键有且只有一个对应的记录。全局索引提供了更强的保证，但是随着表增大，update/delete 操作损失的性能越高，因此更适用于小表。

非全局索引：默认的索引实现，只能保证数据在分区的唯一性。非全局索引依靠写入器为同一个记录的 update/delete 提供一致的分区路径，大幅提高了效率，更适用于大表。从索引的维护成本和写入性能的角度考虑，维护一个全局索引的难度更大，对写入性能的影响也更大，所以需要使用非全局索引。

  • 事实表的延迟更新：例如交易订单，这些表通常会随着时间而不断增长，较少的长尾更新会指向较旧的数据，这要么是由于稍后的交易结算/数据更正所致。换句话说，大多数更新都会进入最新的分区，而很少有更新会进入较旧的分区。适用于 Bloom。
  • 事件表的去重：例如电表度数，事件通常是事实表大小的 10-100 倍，并且通常将“时间”视为一等公民。重复事件大多处于入湖前，且插入和更新仅跨越最后几个时间分区中。适用于 Bloom。
  • 维度表的随机更删：例如商家信息，这些类型的表通常包含高维数据并保存参考数据，更新分布在从旧到新的数据集中的许多分区和数据文件中。通常，这些表无分区。适用于 Simple。
Config Name Default Description
hoodie.index.type N/A (必选) 可选值：SIMPLE、GLOBAL_SIMPLE、BLOOM、GLOBAL_BLOOM、INMEMORY、HBASE、BUCKET、FLINK_STATE、RECORD_INDEX
Spark默认：SIMPLE，FLINK默认：BLOOM，JAVA默认：BLOOM
hoodie.index.class 可选 用户定义索引类的完整路径，并且必须是 HoodieIndex 类的子类。如果指定，它将优先于 hoodie.index.type 配置
Simple Index

核心：利用 Parquet 列式存储的优势，仅需扫描 HooieKey 和 FileID 两列少量的数据，即可用于索引查找。

  1. 对于传入的 Upserts 数据，提取分区路径和主键值，组成 HoodieKey。形成 HoodieKey 到 HoodieRecord的映射。
  2. 从数据湖存储中根据分区路径获取所有 Parquet 文件的 HooieKey 和 FileID 数据。形成 HoodieKey 到 HoodieRecordLocation 的映射。
  3. 将两份数据在对应的分区内，使用 HoodieKey 进行 LeftOuterJoin，即可将提取位置信息赋值至传入数据上。
Bloom Index

核心：使用 Parquet 页脚中的 Bloom 和 Max-Min 数据进行裁剪和过滤减少索引查找时的数据的扫描量。

  1. 对于传入的 Upserts 数据，提取分区路径和主键值，组成 HoodieKey。形成 HoodieKey 到 HoodieRecord的映射。
  2. 从数据湖存储中根据分区路径获取所有 Parquet 文件的页脚数据。页脚数据包含该文件中 HoodieKey 的 Bloom 和最大最小值。
  3. 依据页脚数据中 Hoodiekey 的最大最小值构造线段树，快速的确认传入数据可能存在的 Parquet 文件。
  4. 再利用页脚数据中 Bloom 过滤器，快速的确认传入数据可能存在的 Parquet 文件。
  5. 由于 Bloom 假阳的可能，对匹配到的 Parquet 文件进一步的判断是否存在于该 Parquet 文件，最终形成形成的映射。
  6. 将两份数据在对应的分区内，使用 HoodieKey 进行 LeftOuterJoin，即可将提取位置信息赋值至传入数据上。
Multi-Modal Index

Hudi 在快速更新插入时充分的利用了索引的优势，但在查询尚未利用这些优势。在 Hudi 0.11.0版本中，通过增强元数据表实现了 Multi-Modal Index，以优化查询和写入事务的性能，特别是对于大型的宽表。同其他系统一样，例如 Kafka 将 consumer_offset 存储成一个 topic ，HBase 将表信息存储成一张 meta 表，Hudi 也将具有 Multi-Modal Index 的 MetaData 存储成了内部的一张 MOR 表。目前该表具有 file 、column_stats 、bloom_filter 三个分区。

Config Name Default Description
hoodie.metadata.enable true (Optional) 启用内部元数据表，该表提供表元数据，例如 file 、column_stats 、bloom_filter
hoodie.metadata.index.bloom.filter.enable false (Optional) 在元数据表下启用用户数据文件的索引布隆过滤器。启用后，元数据表将有一个分区来存储布隆过滤器索引，并将在索引查找期间使用。
hoodie.metadata.index.column.stats.enable false (Optional) 在元数据表下启用用户数据文件的索引列范围过滤。启用后，元数据表将有一个分区来存储列范围，并将用于在索引查找期间修剪文件。

file 的主要目的是消除“遍历文件”操作。读写数据时，会执行文件遍历操作来获取文件系统的当前视图，当数据集很大时，列出所有文件可能会成为性能瓶颈，但更重要的是，对于像AWS S3这样的云存储系统，大量的文件列出请求有时会被限制。元数据表将主动维护文件列表，并消除递归文件列表操作的需要。

Config Name Default Description
hoodie.metadata.enable true (Optional) 启用内部元数据表，该表提供表元数据，例如 file 、column_stats 、bloom_filter
hoodie.metadata.index.bloom.filter.enable false (Optional) 在元数据表下启用用户数据文件的索引布隆过滤器。启用后，元数据表将有一个分区来存储布隆过滤器索引，并将在索引查找期间使用。
hoodie.metadata.index.column.stats.enable false (Optional) 在元数据表键查找下启用用户数据文件的索引列范围。启用后，元数据表将有一个分区来存储列范围，并将用于在索引查找期间修剪文件。

column_stats 主要包含了每个文件的列统计信息，例如某一列的 max-min 数据，通过这些统计数据来排除掉不需要读的文件，这样可以极大的提高查询速度。其中每个 RecordKey 是由列名称、分区名称和文件名称组成。

Config Name Default Description
hoodie.metadata.index.column.stats.column.list N/A(Required) 启用列统计信息的列，默认所有的列都会呗索引

bloom_filter 分区取代了 footer 中 的 bloom_filter，在大表中可以大幅减少文件读取的时间。每条记录包含单个数据文件（partition + filename）的 bloom filter，支持 prefix lookup。

Config Name Default Description
hoodie.metadata.index.bloom.filter.column.list N/A(Required) 启用布隆过滤器的列，默认为主键列
